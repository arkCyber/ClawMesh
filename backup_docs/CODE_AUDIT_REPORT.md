#### 📊 并发能力评估

| 指标 | 当前值 | 10万用户需求 | 评估 |
|------|--------|-------------|------|
| 最大并发投递 | 1,000 | 需要 ~5,000 | ⚠️ 需要提升 |
| 批量处理大小 | 100 | 100-500 | ✅ 合理 |
| 在线用户容量 | 100,000 | 100,000 | ✅ 充足 |

**计算依据**:

- 每人平均每分钟发送1条消息
- 峰值并发需求 = 5,000 / 60 ≈ 83 msg/s
- 当前1000并发任务 **足够** 处理

### 2.2 离线消息缓存

**文件**: `crates/clawmesh/messaging/src/offline_cache.rs`

#### ✅ 缓存容量

```rust
const MAX_MESSAGES_PER_USER: usize = 1000;        // 每用户1000条
const MAX_TOTAL_CACHED_MESSAGES: usize = 100_000; // 总共10万条
```

#### 📊 缓存容量评估

| 场景 | 计算 | 结果 |
|------|------|------|
| 10万用户，95%离线 | 95,000 用户 × 平均10条 = 950,000条 | ❌ **不足** |
| 10万用户，50%离线 | 50,000 用户 × 平均10条 = 500,000条 | ❌ **不足** |
| 10万用户，10%离线 | 10,000 用户 × 平均10条 = 100,000条 | ✅ **刚好** |

**⚠️ 关键发现**: 
- 当前缓存容量假设**只有10%用户离线**
- 如果离线率更高，缓存会满
- **建议**: 提升到 `MAX_TOTAL_CACHED_MESSAGES = 1_000_000`

## 3. 性能瓶颈分析

### 3.1 已识别的瓶颈

#### ⚠️ 瓶颈 #1: 内存缓存限制

**位置**: `offline_cache.rs:18`
```rust
const MAX_TOTAL_CACHED_MESSAGES: usize = 100_000;
```

**影响**:
- 10万条消息 × 平均500字节 = 50MB（可接受）
- 但如果需要缓存100万条 = 500MB（需要考虑）

**建议**:
```rust
// 根据可用内存动态调整
const MAX_TOTAL_CACHED_MESSAGES: usize = 1_000_000; // 提升10倍
```

#### ⚠️ 瓶颈 #2: 单点写入锁

**位置**: `offline_cache.rs:265`
```rust
let mut queues = self.queues.write().await; // 写锁会阻塞所有读取
```

**影响**:
- 高并发写入时会产生锁竞争
- 可能导致延迟增加

**建议**:
```rust
// 使用分片锁 (Sharded Lock)
// 将用户ID哈希到16个分片，减少锁竞争
const SHARD_COUNT: usize = 16;
queues: Arc<Vec<RwLock<HashMap<i32, UserMessageQueue>>>>
```

#### ⚠️ 瓶颈 #3: 缺少数据库持久化

**位置**: `delivery.rs:146`
```rust
// TODO: Implement with database integration
```

**影响**:
- 服务重启后离线消息丢失
- 不符合航空航天级别可靠性标准

**建议**: 实现消息持久化到数据库

### 4.2 缺少的限制

#### ❌ 缺少速率限制

**位置**: `direct_message.rs:125`
```rust
// - Rate limit exceeded  // 注释提到但未实现
```

**风险**: 用户可以无限发送消息，导致：
- 垃圾消息泛滥
- 系统资源耗尽
- DoS攻击风险

**建议实现**:
```rust
// 每用户每分钟最多60条消息
const MAX_MESSAGES_PER_MINUTE: u32 = 60;

// 使用 Token Bucket 或 Sliding Window 算法
pub struct RateLimiter {
    limits: Arc<RwLock<HashMap<i32, TokenBucket>>>,
}
```

#### ❌ 缺少数据库连接池配置

**风险**: 数据库连接耗尽

**建议**:
```rust
// 数据库连接池配置
const DB_POOL_SIZE: u32 = 100;  // 100个连接
const DB_MAX_OVERFLOW: u32 = 50; // 最多溢出50个
```
